## Deferred from: code review of story-1-1 (2026-06-22)

- **TOCTOU port race** [src/dev_helper_mcp/server.py:18-44] â€” `find_free_port` binds+closes a probe socket, then `uvicorn.run` re-binds the same port by number. Between the probe close and the uvicorn bind another process can grab the port, crashing with an unhandled `OSError` (no retry/fallback). `SO_REUSEADDR` on the probe widens the gap. Robust fix: hand the already-bound socket to uvicorn (`uvicorn.Server` with a passed socket) instead of re-binding by number. Deferred to Story 3.1 (single-instance protection / lockfile); low real-world impact for a localhost single-user tool.

## Deferred from: code review of story-1-3 (2026-06-22)

- **Post-preflight partial state / no compensation** [src/dev_helper_mcp/core/tasks.py:116-133] â€” once provisioning starts, a `worktree add` failure on repo N (race, disk error) leaves repos 1..N-1's worktrees + `agent/<slug>` branches orphaned on disk with no DB row and no teardown; the failure also surfaces as a generic `Internal` rather than typed `BranchExists`/`WorktreePathInUse` even when the cause is a post-preflight collision. Story 1.3 is happy-path + preflight only â€” full reverse-order compensation, `RollbackIncomplete`, and the `RepoMutator` fault-injection seam are explicitly Story 1.4. Deferred to Story 1.4.
- **Per-repo mutex is process-local** [src/dev_helper_mcp/core/tasks.py:63-70] â€” the `RepoLockRegistry` serializes only coroutines within one process; two server processes pointing at the same repos/state would not serialize, and even within one process two same-slug creates over *disjoint* repo sets share no lock, so both can pass the `status != 'done'` gate and provision worktrees before the persist-time `ActiveTaskConflict` rejects the loser (orphaning its worktrees). Mitigated by the machine-global single-instance lockfile (Story 3.1, which enforces one server process) + post-failure rollback (Story 1.4). Deferred.

## Deferred from: code review of story-1-4 (2026-06-23)

- **`except DevHelperError` doesn't catch `CancelledError`/`BaseException`** [src/dev_helper_mcp/core/tasks.py:143] â€” the provisioning try/except only triggers compensation on a `DevHelperError`. A task cancellation (server shutdown, client disconnect) or any non-typed exception raised mid-`mutator.add` propagates without running the reverse-order teardown, leaking already-provisioned worktrees + `agent/<slug>` branches. In production `add`/`remove` only raise `DevHelperError` (verified: `run_git` raises only `GitTimeout`/`NotAGitRepo`/`Internal`, all `DevHelperError`), so the live exposure is `asyncio.CancelledError`. Adjacent to the explicit v1 crash-safety non-goal; a correct fix must `asyncio.shield` the teardown so cancellation doesn't abort it mid-rollback. Deferred â€” hardening, non-trivial.
- **`_classify_add_failure` is locale-fragile** [src/dev_helper_mcp/core/mutator.py:528-546 / src/dev_helper_mcp/git/runner.py:132] â€” typed classification of a non-zero `git worktree add` matches English stderr substrings (`"a branch named"`, `"is already checked out"`, `"already exists"`). The runner env (runner.py:132) strips `GIT_*` context vars and pins `GIT_TERMINAL_PROMPT`/`GIT_OPTIONAL_LOCKS` but does NOT pin `LC_ALL=C`/`LANG=C`, so under a non-English system locale git's localized stderr falls through to the generic `Internal`. Degrades safely (compensation still runs; story sanctions `Internal` as the best-effort fallback), but an agent branching on the stable typed code mis-handles the collision. Fix: pin `LC_ALL=C` in the runner env (also hardens all future porcelain parsing) â€” edits `runner.py`, which Story 1.4 holds unchanged, hence deferred.
- **`remove`'s `branch -D` force-deletes unguarded at the reusable seam boundary** [src/dev_helper_mcp/core/mutator.py:600] â€” `GitRepoMutator.remove` unconditionally runs `git branch -D <branch>`. Safe in 1.4 (the branch was just created by this very call, so it is brand-new + unmerged and `-D` is correct), but the module docstring explicitly anticipates Story 1.5 extending/reusing this seam for the user-facing `remove_worktree` tool. A future caller invoking `remove` on a pre-existing branch would silently force-delete unmerged work. Guard the destructive `-D` at the API boundary (or rename to a teardown-only method) before Story 1.5 reuses it. Deferred to Story 1.5.
